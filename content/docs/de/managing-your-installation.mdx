Diese Seite deckt den laufenden Betrieb einer self-hosted Customermates-Installation ab.
Wenn du den Stack noch nicht installiert hast, beginne mit [Self-Hosting Get Started](/docs/self-hosting).

Der self-hosted Betrieb nutzt die Community Edition. Kostenpflichtige Enterprise-Features (Audit Logging, Single Sign-On, AI Agent, Whitelabeling) sind nur mit expliziter Lizenz und passendem Deployment-Modell verfügbar.

## Installation aktualisieren

Im Deployment-Verzeichnis:

```bash
chmod +x scripts/selfhost-update.sh
./scripts/selfhost-update.sh
```

Optional kannst du vor dem Neubau die neuesten Git-Aenderungen ziehen:

```bash
./scripts/selfhost-update.sh --pull
```

Das Update-Skript:

1. optional die neuesten Repository-Aenderungen holen
2. das App-Image neu bauen
3. `prisma migrate deploy` ausfuehren
4. den App-Service neu starten

Nach jedem Update solltest du den Status mit `docker compose ps` pruefen.

## Restart-Flow

Verwende den Restart-Helper, wenn Konfigurationsaenderungen einen sauberen Neustart erfordern:

```bash
chmod +x scripts/selfhost-restart.sh
./scripts/selfhost-restart.sh
```

Dieses Skript sollte den Stack sicher neu starten und Hinweise zu Health-Checks und Logs ausgeben.
Es fuehrt keine Migrationen aus.

## Logs und Troubleshooting-Grundlagen

Viele Betriebsprobleme lassen sich ueber Container-Logs eingrenzen:

```bash
docker compose logs -f app
docker compose logs -f postgres
```

Nuetzliche Checks:

- `docker compose ps` fuer Status und Health der Services
- `docker compose config` zur Validierung der zusammengefuehrten Compose-Konfiguration
- `docker compose exec app sh` zur Laufzeitinspektion im App-Container

## Haeufige Docker Compose Befehle

Typische Operationen im Alltag:

```bash
docker compose up -d
docker compose down
docker compose pull
docker compose build --no-cache app
docker compose restart app
```

Gezielte Restarts (`docker compose restart app`) sind sinnvoll, wenn nur die Anwendung geaendert wurde.

## Alle Self-Hosted-Daten zuruecksetzen

Verwende den Reset-Helper nur, wenn du lokale Deploy-Daten bewusst loeschen willst:

```bash
chmod +x scripts/selfhost-reset.sh
./scripts/selfhost-reset.sh
```

Das Skript verlangt eine explizite Bestaetigung (`RESET`), bevor Volumes geloescht werden.

## Optionale Betriebs-Hinweise

Fuer produktive Deployments sollte es einen Backup- und Restore-Prozess geben:

- sichere PostgreSQL-Daten regelmaessig
- teste Restore-Prozesse in einer Nicht-Produktionsumgebung
- halte Umgebungsdateien und Secrets aus der Versionskontrolle heraus

Wenn du erst mit Self-Hosting startest, gehe zur [Self-Hosting Schnellstart-Seite](/docs/self-hosting).
