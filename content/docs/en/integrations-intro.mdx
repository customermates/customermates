**TL;DR** — MCP for AI-driven read/write. OpenAPI for engineered integrations. Webhooks for push notifications of change. n8n for visual workflows. Most teams use several at once.

## Decision matrix

| I want to… | Use |
|---|---|
| Have Claude/ChatGPT update my CRM in conversation | [MCP](/docs/mcp) |
| Build a custom integration in code | [OpenAPI](/docs/openapi) |
| React to CRM changes in my stack (Slack, Sequencer, BI) | [Webhooks](/docs/webhooks) |
| Compose workflows visually without code | [n8n](/docs/n8n) |
| Sync two-way with a Zapier-heavy stack | [Webhooks](/docs/webhooks) + REST |

## MCP

Model Context Protocol endpoint at `/api/v1/mcp`. Your AI client discovers 57 CRM tools, calls them by name, and gets structured errors back. Designed to work reliably even with weak models through narrow schemas, inline enum hints, and relationship safety rules.

[**MCP overview →**](/docs/mcp) · [**Tool catalog →**](/docs/mcp-tool-catalog)

## REST (OpenAPI 3.1)

Canonical HTTP surface. Every CRM operation is a documented endpoint with request and response schemas. Use this when you are writing integration code and you already know what you need to call.

[**OpenAPI reference →**](/docs/openapi)

## Webhooks

Every entity write emits a domain event. Subscribe by URL, we POST JSON there on every change. Signed with HMAC-SHA256 if you set a secret. Deliveries are logged and re-sendable.

[**Webhooks →**](/docs/webhooks) · [**Event catalog →**](/docs/webhook-events)

## n8n

For teams that prefer visual automation. The Customermates n8n nodes wrap the MCP and REST surfaces with drag-and-drop triggers and actions.

[**n8n →**](/docs/n8n)

## Which combination is right?

- **Solo operator with an AI-first workflow:** MCP for writes, webhooks into Slack or email for notifications. No REST needed.
- **Small sales team:** Webhooks into their existing sequencer, MCP for the founder's AI, occasional OpenAPI scripts for data cleanup.
- **Integrator building on top of Customermates:** OpenAPI for the bulk, webhooks for event-driven work, MCP if your product is itself agent-driven.
